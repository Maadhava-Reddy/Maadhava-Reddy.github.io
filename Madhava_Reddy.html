PROJECT 01 · SYSTEMS / NETWORKING
NetPulse
Multi-threaded TCP chat server · C++ · Linux concurrency
GitHub ↗
Problem

A chat server has to hold hundreds of live connections without a thread per client eating the machine alive. NetPulse answers with an event-driven core: one epoll loop watching every socket, a worker pool doing the actual message work.

Impact
200+
concurrent clients handled
Sanitizer-clean
ASan/UBSan build targets
Race-free
debugged with GDB & strace
Challenges & Decisions
Challenges

Keeping the epoll loop and the worker pool honest with each other — shared connection state under concurrency is where subtle races live. Debugging meant instrumented builds, GDB sessions, and strace-ing syscall behavior under load.

Decisions
  • epoll event loop over thread-per-client — scales connections without scaling threads.
  • ASan/UBSan targets baked into the Makefile so memory bugs surface in dev, not production.
Architecture
Client Sockets↓ epoll Event Loop↓ Worker Thread Pool↓ Message Broadcast
Stack
C++LinuxepollpthreadsMakeASan / UBSanGDBstrace
PROJECT 02 · IOT / AI
Smart Water Tank Monitoring
IoT + AI/ML · ESP32 firmware to live dashboard · built at NIT Warangal
GitHub ↗
Problem

Water tanks are usually managed blind — nobody knows the level, the flow, or that a leak started until it's a problem. This system gives a tank a nervous system: live sensing, a backend that remembers, and ML that flags when consumption looks wrong.

Impact
Real-time
level, flow & temperature sensing
Predictive
consumption analysis on live data
Sensor→Screen
firmware-to-dashboard pipeline
Challenges & Decisions
Challenges

Bridging two very different worlds: memory-constrained embedded firmware pushing readings upstream, and a data backend that has to treat noisy, real-world sensor streams as something an anomaly model can actually learn from.

Decisions
  • FastAPI + PostgreSQL as the backbone — typed, fast APIs with a relational store built for time-stamped sensor history.
  • Anomaly detection runs server-side, keeping ESP32 firmware lean and models easy to retrain.
Architecture
ESP32 + Sensors↓ C/C++ Firmware↓ FastAPI + PostgreSQL↓ Live Dashboard + ML Anomaly Detection
Stack

PROJECT 03 · AI / SECURITY
APK Secure
Android malware detection · Python static analysis + machine learning
GitHub ↗
Problem

Malicious Android APKs are easy to disguise as legitimate apps, and most users have no fast way to check a file before installing it. APK Secure inspects an APK statically and lets an ML classifier call it: clean or malware.

Impact
Static
no execution needed — safe to analyze
ML-scored
permission & API-level features
Validated
modeling & validation first-class
Challenges & Decisions
Challenges

Extracting meaningful signal from APK metadata and permissions without full binary disassembly, and tuning the classifier so false positives stay low without letting real threats slip through.

Decisions
  • Static features over dynamic sandboxing — faster, safer, and enough signal for a strong classifier.
  • Validation treated as part of the system: hold-out evaluation before trusting any verdict.
Architecture
APK Input↓ Static Feature Extraction↓ ML Classifier↓ Risk Verdict
Stack
